# ui-01 — Append-only grid + debounced, capped persistence

**Commit:** `e59502b` — "Web UI: O(1) per-token grid append + debounced
capped persistence (was O(T) rebuild + O(T) stringify per token)".
**Code:** `web/index.html` (`renderGrid`, `buildGridRowTr`, `saveState`).

## The problem (measured, headless)

Once decoding was fast (decode-01/02), the browser became the bottleneck.
The J-Space grid has **64 layer columns + 1 token-label column** and **one
row per generated token**. Profiling a real stream (~294 rows, 19,110
`<td>` cells) with instrumented render/persist calls found:

| Component | mean | trend over the stream | total |
|-----------|------|----------------------|-------|
| `renderGrid` (full table rebuild per token) | 93 ms | 18 ms → **180 ms**, growing | 24.5 s |
| `saveState` (stringify *all* snapshots → localStorage) | 18 ms | 2.5 → 30 ms | 4.8 s |

So the UI burned ~150–210 ms/token and climbing — O(T²) overall. A side
finding: the localStorage blob had already hit **5.1 MB**, right at the
quota, so persistence was silently failing (70 console warnings).

## The fix

- **Append-only grid while streaming:** extract `buildGridRowTr`; when
  streaming with nothing selected, append only the new bottom rows instead
  of rebuilding the whole table. Full rebuild only on selection / reload.
- **Debounced persistence:** `saveState` debounced to 1 s, flushed at stream
  end and `beforeunload`; scores rounded to 2 dp at ingest; the persisted
  blob capped at 4.2 MB by dropping the oldest snapshots (previously it
  silently persisted *nothing* at quota).

## Result

`renderAll` dropped from **85–180 ms/token growing** to **5.4 ms/token
flat** — ~17× less work, no degradation with length; total UI main-thread
time per stream 25 s → 0.7 s. Cell selection, detail panel, and deselection
all verified working headlessly.

## Note

This was superseded by ui-02 (virtualization), which is strictly better:
append-only still kept *all* rows in the DOM, which ui-02 fixes.
